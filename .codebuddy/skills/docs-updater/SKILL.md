# 📚 文档更新器

在代码变更后同步更新相关文档，确保文档与代码保持一致。

## 🎯 更新目标

**仅更新 docs 目录下对外的功能文档**。

**不需要更新文档的情况**：
- 性能优化
- 代码重构
- 脚手架变动
- 内部实现调整
- 测试相关变更

**需要更新文档的情况**：
1. **识别影响范围**：根据代码变更确定是否影响对外功能
2. **检查文档存在**：查找 docs 目录下的相关文档
3. **更新内容**：确保文档与对外功能保持一致
4. **添加新文档**：如果新增对外功能需要文档但不存在，创建新文档

## 📋 检查步骤

### 1. 查找 docs 目录
```bash
ls -la docs/ 2>/dev/null || echo "docs 目录不存在"
```

### 2. 列出文档文件
```bash
find docs -type f -name "*.md" 2>/dev/null | head -20
```

## 🎯 文档关联规则

**只有影响对外功能的变更才需要更新文档**：

| 变更类型 | 是否更新 | 相关文档 |
|---------|---------|----------|
| 新增 CLI 命令 | ✅ 需要 | CLI 使用文档、命令参考 |
| 修改配置选项 | ✅ 需要 | 配置文档、示例配置 |
| 新增 API | ✅ 需要 | API 文档、集成指南 |
| 修改工作流程 | ✅ 需要 | 用户指南、快速开始 |
| 性能优化 | ❌ 不需要 | - |
| 代码重构 | ❌ 不需要 | - |
| 脚手架变动 | ❌ 不需要 | - |
| 内部实现调整 | ❌ 不需要 | - |
| 测试相关 | ❌ 不需要 | - |

## 📝 常见文档位置

- `docs/README.md` - 文档索引
- `docs/guide/` - 用户指南
- `docs/api/` - API 文档
- `docs/cli/` - CLI 命令文档
- `docs/config/` - 配置文档
- `docs/examples/` - 示例代码

## 🎯 更新策略

| 场景 | 操作 |
|------|------|
| **功能新增** | 在相关文档中添加新章节、更新目录索引、添加使用示例 |
| **功能修改** | 更新描述和参数说明、修改示例代码、标注版本变更 |
| **功能移除** | 添加弃用说明、标注移除版本、提供替代方案 |

## 📝 文档规范

### 中文文案排版规范

遵循 [中文文案排版指北](https://github.com/sparanoid/chinese-copywriting-guidelines) 规范。

**核心规则**：

| 规则 | ✅ 正确 | ❌ 错误 |
|------|--------|--------|
| **中英文之间需要空格** | 在 LeanCloud 上 | 在LeanCloud上 |
| **中文与数字之间需要空格** | 支持 10 个并发 | 支持10个并发 |
| **数字与单位之间需要空格** | 10 Gbps | 10Gbps |
| **使用全形中文标点符号** | 你好！ | 你好! |
| **专有名词使用正确的大小写** | GitHub, TypeScript | github, typescript |
| **全形标点与其他字符之间不加空格** | 使用 GitHub，可以 | 使用 GitHub ，可以 |

**标题层级**：
```markdown
# 一级标题 - 文档主题
## 二级标题 - 主要章节
### 三级标题 - 子章节
```

**代码示例**：
````markdown
```bash
codebuddy --help
```
````

**表格格式**：
```markdown
| 参数 | 类型 | 说明 |
|------|------|------|
| name | string | 名称 |
```

## 输出要求

**判断逻辑**：
1. 首先判断本次变更是否影响对外功能
2. 如果是内部变更（性能优化、代码重构、脚手架变动等），直接说明「本次为内部变更，无需更新文档」
3. 如果影响对外功能，则报告已更新的文档和建议创建的文档
